System Architecture and Implementation Plan for a University-Aware Academic Study AssistantThe integration of university-specific academic context into a study web application transforms the user experience from a generic task manager into an intelligent, curriculum-aware academic operating system. This specification details the design, data architecture, web scraping strategy, document ingestion pipelines, data fusion logic, and execution plan required to implement a robust, automated academic onboarding and personalization engine.Feature Definition and Core Product ValueThe university-aware academic study assistant functions as a localized integration layer within the study web application, dynamically adapting the platform’s interface, notification schedules, task recommendations, and search utilities based on a student’s specific institutional profile. Rather than relying on tedious manual entry of course schedules and academic dates, the system automates this process by systematically crawling and parsing official university portals, department catalogs, and course syllabi.Once initialized, the platform operates as a personalized academic operating system. The calendar is configured to display accurate institutional terms, and the study recommendation engine dynamically adjusts study intensity based on the student's current position within the academic semester. For example, the system automatically escalates study priorities prior to midterm and final exam blocks while suggesting shifts in focus during institutional breaks or add/drop windows.User Flow and Onboarding OrchestrationThe onboarding interface is designed to capture the student's exact academic position while maintaining a frictionless user experience. The collection flow is structured around progressive disclosure, where each selection dynamically constrains subsequent dropdown options to ensure data integrity and prevent institutional classification errors.The interaction timeline is divided into five sequential phases:StepUser ActionSystem ProcessingUI Feedback & State Management1. Standard text extraction tools disrupt column readings and render tabular schemas illegible, which corrupts downstream retrieval systems. This conversion strategy employs specific pipelines to extract structured data from academic documents:Pythonimport s3fs
from docling.document_converter import DocumentConverter
from docling.datamodel.base_models import InputFormat
from docling.datamodel.pipeline_options import PdfPipelineOptions
from llama_index.core.node_parser import SentenceSplitter

# Configuration of the layout-aware parser
pipeline_options = PdfPipelineOptions()
pipeline_options.do_ocr = True  # Enable OCR for scanned syllabi
pipeline_options.do_table_structure = True  # Enable TableFormer reconstruction

converter = DocumentConverter(
    allowed_formats=[InputFormat.PDF, InputFormat.DOCX],
    pipeline_options=pipeline_options
)

def ingest_academic_document(s3_uri: str, metadata: dict) -> list:
    """
    Parses unstructured academic documents into layout-aware, vectorized text nodes.
    """
    # Fetch file from S3 bucket [cite: 9]
    fs = s3fs.S3FileSystem()
    with fs.open(s3_uri, "rb") as file_stream:
        conversion_result = converter.convert(file_stream)
        
    # Export structural hierarchical Markdown representation
    markdown_content = conversion_result.document.export_to_markdown()
    
    # Segment document preserving section boundaries
    splitter = SentenceSplitter(chunk_size=512, chunk_overlap=64)
    text_chunks = splitter.split_text(markdown_content)
    
    nodes = []
    for index, chunk in enumerate(text_chunks):
        node_metadata = {
            **metadata,
            "chunk_index": index,
            "length": len(chunk)
        }
        nodes.append({
            "text": chunk,
            "metadata": node_metadata
        })
    return nodes
The layout-aware parsing pipeline ensures that structured academic documents are broken down into contextual chunks while maintaining metadata associations. This systematic indexation directly solves the four key user query flows within the RAG pipeline:Query 1: "When do finals start?"Ingested Source: Institutional Academic Calendar PDF, Registrar Announcements.Indexation Logic: Extracts absolute term milestones using structural mapping, parsing ranges labeled Final Examination Period into dedicated scalar date objects.Query 2: "What courses are in my first-year curriculum?"Ingested Source: Department Curriculum Handbook, Course Catalog PDF.Indexation Logic: Preserves hierarchical relationships of degree pathways, grouping courses by Year and Semester levels within the curriculum maps.Query 3: "What are the important academic deadlines this month?"Ingested Source: Calendar schedules, drop/add deadlines, academic break announcements.Indexation Logic: Identifies temporal variables across documents, using chronological indexes to query events falling within the active month's date bounds.Query 4: "What should I focus on this week based on my semester schedule?"Ingested Source: Active syllabi, course outlines, exam timelines.Indexation Logic: Resolves course codes, matches structural timelines across active enrollments, and surfaces assignments matching the current academic week.Data Verification, State Management, and Conflict ResolutionBecause academic data can be pulled from several sources (e.g., student uploads, department web pages, and central registrar portals), information must be validated to prevent inconsistencies. The system enforces three data state classifications:Confirmed Official Information: Data extracted directly from Tier 1 central sources (e.g., the university registrar's domain) or verified by a administrator, carrying a confidence score of $\ge 0.95$.Inferred Academic Structure: Information constructed from secondary sources, such as syllabus files, departmental newsletters, or course schedule drafts, showing a confidence score of $0.60 \le P < 0.95$.Missing Data: Schedules or timelines that are completely unverified, unpopulated, or return a confidence score of $< 0.60$, forcing the system to fall back to a transparent empty-state indicator.Bayesian Conflict Resolution ArchitectureWhen multiple sources report conflicting dates or course attributes, the system applies a Bayesian consensus model to resolve discrepancies without manual intervention. The credibility score of an individual source $S_i$ is calculated dynamically as:$$W(S_i) = \alpha \cdot \text{BaseTier}(S_i) + \beta \cdot \text{ConsistencyHistory}(S_i) - \gamma \cdot \text{DecayRate}(\Delta t)$$$\text{BaseTier}(S_i)$ maps to the authority tier of the domain.$\text{ConsistencyHistory}(S_i)$ reflects the historical ratio of validated vs. rejected dates sourced from that domain.$\text{DecayRate}(\Delta t)$ reduces the credibility of sources that have not been updated recently.For any target academic date field $d_k$, let $V_k = \{v_{k1}, v_{k2}, \dots, v_{kn}\}$ be the set of unique values reported by various sources. The system computes the consensus probability for each value $v_{kj}$:$$P(d_k = v_{kj} \mid \mathbf{S}) = \frac{\exp\left(\sum_{S_i \in \mathbf{S}} W(S_i) \cdot \mathbb{I}(S_i \text{ reports } v_{kj})\right)}{\sum_{v' \in V_k} \exp\left(\sum_{S_i \in \mathbf{S}} W(S_i) \cdot \mathbb{I}(S_i \text{ reports } v')\right)}$$where $\mathbb{I}$ is an indicator function evaluating to 1 if source $S_i$ asserts value $v_{kj}$, and 0 otherwise.Resolution Logic HierarchyIf a conflict occurs, the system applies the following prioritized rules:Rule 1: Recency and Centrality Dominance. Central Registrar entries (Tier 1) override local department catalogs or syllabi (Tiers 2-4) for institutional milestones (e.g., add/drop and withdrawal deadlines).Rule 2: Specificity Allocation. If a course syllabus (Tier 4) lists a class quiz date that differs from the general department newsletter (Tier 3), the syllabus date is selected for that specific class.Rule 3: Transparent State Management. The application displays dates with clear visual indicators to communicate verification confidence:[Oct 12] Add/Drop Deadline  --->  [ Verified ]  (Sourced from registrar.uw.edu)
[Nov 04] Midterm Exam Date  --->  [ Inferred ]  (Sourced from Syllabus Upload)
[Dec 15] Final Exam Date    --->  [ Unreleased] (No official exam schedule published yet)
The system explicitly refuses to invent dates. If an exam schedule is not found in Tier 1 or Tier 2 sources, it is set to NULL in the database, and the user-facing interface shows an "Unreleased" status instead of guessing.Academic-Aware Planner and System IntegrationThe platform integrates verified academic timelines directly into its scheduling algorithms and recommendations, adapting to the student's actual cognitive workload.Planner Workload Calculations and Priority ShiftsThe planner estimates a student's cognitive load curve to optimize scheduled study hours. The target cognitive load $L_s(t)$ for student $s$ at time $t$ is calculated across all active course enrollments $C_s$:$$L_s(t) = \sum_{c \in C_s} \sum_{e \in E_c} \frac{\omega(e)}{\left(\frac{T(e) - t}{\delta}\right)^\lambda + \epsilon}$$$E_c$ is the set of all active exam events, homework due dates, or academic milestones for course $c$.$\omega(e)$ is the predefined intensity weight of event $e$ (e.g., final exam = 1.0, midterm = 0.7, weekly homework = 0.2, optional reading = 0.05).$T(e)$ represents the target datetime of event $e$.$\delta$ is a scaling factor normalizing temporal delta measurements into days (default = 1.0).$\lambda$ is a decay power curve modeling the exponential stress increase as a deadline approaches (default = 1.5).$\epsilon$ is a smoothing constant preventing division-by-zero errors at the exact moment of the event.Based on the calculated $L_s(t)$, the system scales study schedules and triggers priority shifts:  Cognitive Load (L)
