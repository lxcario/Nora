# Cross-Topic "Eureka" Connections — Requirements

## Introduction

Eureka Connections surfaces surprising inter-subject links between concepts the student is studying. Using embeddings to find cross-topic semantic similarity, the system presents "eureka moments" — unexpected parallels between different subjects. Each connection includes a challenge: explain how the concepts relate. Successful explanations earn bonus XP. The feature runs as a background job (not real-time) and presents findings as delightful thought-bubble animations from the pet.

**Scope boundary:** Eureka Connections is a passive discovery system, not an active study mode. It produces 1–3 connections per day (or on new Feynman submission). It does not create new topics, cards, or notes — it only surfaces observations for the student to optionally engage with. The "explain" challenge uses the existing Feynman evaluation pipeline.

## Glossary

- **Eureka_Connection**: A discovered semantic relationship between two topics/concepts from different subjects, presented as an insight.
- **Eureka_Challenge**: A prompt asking the student to explain how two connected concepts relate, evaluated by the Feynman pipeline.
- **Eureka_XP**: Bonus XP awarded for successful challenge explanations.
- **Connection_Engine**: The server-side module that discovers cross-topic connections using embeddings.
- **Eureka_Notification**: A pixel-art thought-bubble animation above the pet announcing a new connection.

## Requirements

### Requirement 1: Connection Discovery

**User Story:** As a student, I want the system to find surprising connections between my subjects so I can see the bigger picture.

#### Acceptance Criteria

1. THE Connection_Engine SHALL compare topic descriptions/card content across different subjects to find semantic similarity using existing embeddings.
2. THE Connection_Engine SHALL only surface connections between topics from DIFFERENT subjects (not within the same subject — those are obvious).
3. THE Connection_Engine SHALL require a minimum similarity threshold of 0.5 (cosine) to consider a connection valid.
4. THE Connection_Engine SHALL filter out connections that are trivially obvious (same keywords, near-duplicate phrasing) by rejecting pairs with similarity > 0.9.
5. THE Connection_Engine SHALL produce at most 3 new connections per day per user.
6. THE Connection_Engine SHALL not produce duplicate connections — once a pair has been shown, it SHALL not be surfaced again.
7. THE Connection_Engine SHALL run as a background computation triggered daily or on new Feynman submission — not on every page load.

### Requirement 2: Connection Presentation

**User Story:** As a student, I want connections presented as delightful "eureka" moments from my pet, not as dry notifications.

#### Acceptance Criteria

1. WHEN a new Eureka_Connection is available, THE system SHALL display a thought-bubble pixel animation above the pet character in the pixel room.
2. THE thought bubble SHALL contain a brief connection phrase (e.g., "Did you notice [Concept A from Physics] uses the same structure as [Concept B from Economics]?").
3. THE connection phrase SHALL be generated by the LLM, limited to 100–200 characters, and include both topic names and a hint of the relationship.
4. THE student SHALL be able to dismiss the notification or click to view the full connection detail.
5. WHEN the student clicks the notification, THE system SHALL display a connection card with: both topic names and subjects, a 1–2 sentence explanation of the similarity, and the challenge prompt.
6. THE system SHALL display an "Explain It" button to start the Eureka_Challenge.
7. WHEN multiple connections are pending (unviewed), THE system SHALL show them one at a time, oldest first.

### Requirement 3: Eureka Challenge

**User Story:** As a student, I want to earn bonus XP by explaining how two concepts are related.

#### Acceptance Criteria

1. WHEN the student clicks "Explain It" on a connection, THE system SHALL prompt them to write an explanation (50–500 characters) of how the two concepts relate.
2. THE system SHALL evaluate the explanation using the existing Feynman evaluation pipeline, with both topic descriptions as context/source reference.
3. WHEN the evaluation score is 60 or above (out of 100), THE system SHALL award Eureka_XP (25 XP, 10 coins) and mark the challenge as completed.
4. WHEN the evaluation score is below 60, THE system SHALL display the feedback (gaps, suggestions) and allow one retry.
5. THE student SHALL be able to skip the challenge without penalty.
6. COMPLETED challenges SHALL be viewable in a history (last 20 connections).

### Requirement 4: Background Job Logic

**User Story:** As a student, I want connections to appear naturally without me having to trigger them.

#### Acceptance Criteria

1. THE Connection_Engine SHALL run automatically once daily (on first page load after 24h since last run).
2. THE Connection_Engine SHALL also run when a new Feynman explanation is submitted (debounced — at most once per 6 hours).
3. WHEN the student has fewer than 2 topics across different subjects, THE Connection_Engine SHALL skip computation and produce no connections.
4. THE system SHALL store discovered connections in the database for later presentation (even if the student hasn't viewed them yet).
5. THE system SHALL retain at most 50 connections per user (rolling window, oldest dismissed connections are deleted).

### Requirement 5: Data Model

**User Story:** As a developer, I want a clean schema for storing discovered connections and challenge results.

#### Acceptance Criteria

1. THE system SHALL store connections in an `eureka_connections` table with columns: id, user_id, topic_a_id, topic_b_id, similarity_score, connection_phrase, status (pending/viewed/completed/dismissed), xp_awarded, created_at.
2. THE `eureka_connections` table SHALL have a UNIQUE constraint on (user_id, topic_a_id, topic_b_id) to prevent duplicate connections.
3. RLS policies SHALL restrict access to the user's own connections.
4. THE system SHALL record challenge attempts (explanation text, score, created_at) in a JSONB `attempts` column on the connection record.
